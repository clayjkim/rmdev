# RM360 개발 일지

> 한국 의료기관을 위한 두 축의 소프트웨어를 개발하고 있습니다.

> ② **통화보안·기록 앱** — 법인폰 통화를 자동으로 안전하게 보관하고 내용을 정리·분석

## 2026-06-09 (화)
- **평판 시스템 — 대응관리 "실사용" 단계 진입**: 악성 게시물을 실제로 내리는 작업을 본격화. 채널마다 신고·삭제 방법이 전부 달라서(바비톡·강남언니·디시인사이드·성예사·네이버·카카오·구글·인스타 등), **채널별 대응 '레시피'를 구조화한 데이터베이스를 신설** — 어디로 어떻게 접수하는지, 본인확인 필요 여부, 필요 서류, 글을 특정하는 항목, 단계별 절차, 법적 근거까지 한곳에. 12개 채널의 실제 접수 경로를 조사·교차검증해 채웠다. 지금은 전부 **사람이 직접 처리**하는 것으로 시작하고, 채널 하나씩 자동화로 전환할 수 있게 모듈식으로 설계.
- **병원 서류 보관함**: 사업자등록증·위임장·명함을 한 번만 올려두면 대응할 때마다 재사용.
- **즉시 알림**: 고객이 '대응'을 요청하거나 새 문제글을 신고하면 담당자에게 **곧바로 이메일 알림**이 가도록 구축 — 누락 없이 즉시 대응 착수. 문자 알림도 나중에 끼울 수 있게 자리 마련.
- **관리자 처리 화면 설계**: 들어온 대응 요청을 한눈에 보는 목록 + 그 채널의 대응 설명서(레시피)를 띄워 따라 처리 + 진행 상태를 기록·누적 + 채널별 대응법을 직접 다듬는 관리 화면.
- **(2차) 대응 화면 사용성 개선 + 공문 발급 안정화**: 실제로 써보니 불편한 점들을 정리. ① 대응 상세를 작은 팝업이 아니라 **독립된 페이지**로 열리게 ② 상단에 진행 단계(접수→검토→준비→전달완료→종결)를 띄우고 **단계를 누르면 바로 상태가 바뀌게**(번거로운 선택 메뉴 제거) ③ 처리 메모를 **여러 개·여러 종류로 계속 쌓고 수정**할 수 있게 ④ 채널마다 연락 방법과 대응 메모를 상세히 적어 관리하는 **채널관리 화면** 추가. 그리고 **정보삭제요청서(공문) PDF가 다운로드되지 않던 문제를 근본 해결** — 별도 서버 작업이 멈추면 같이 멈추던 구조를 걷어내고, 버튼을 누르면 **항상 즉시 발급**되도록 방식을 바꿨다(법률 양식·한글 서식 그대로 재현 검증 완료).

## 2026-06-08 (월)
- **평판 시스템 — 분류 정확도 핵심 개선**: 한 게시글에 긍정·부정 답변이 섞여 있을 때, 긍정이 더 많다는 이유로 부정이 묻혀 '긍정'으로 잘못 분류되던 문제를 발견하고 해결. 이제 **부정 신호가 하나라도 있으면 '부정'으로 우선** 분류한다(평판 관리에서는 부정을 놓치는 것이 가장 위험하므로). 검색결과(지식iN 등)가 URL 차이로 중복 수집되던 것도 표준화로 차단.
- **통화보안 앱 (v0.9.5 → v0.9.6)**: 통화 종료 후 기록 알림을 무시하면 5분 뒤 한 번 더 알림 / 연락처에 저장된 번호는 이름 자동 입력 / 놓친 통화는 나중에 목록에서 일괄 입력(미입력 건수 표시) / 자동 보관 정지 시 경고·앱 강제종료 시 자동 재시작(직원 보호) / 분류를 버튼식으로(클레임 강조) / 새 앱 아이콘 적용.

## 2026-06-07 (일)
- **결제·멤버십 시스템**: PortOne 기반 구독 결제 + 멤버십 관리 + 운영 핸드북 구축.
- **통화보안 앱 (v0.9 완성형 MVP)**: 자동 수집 + 직원 인증 + 통화종료 기록까지 완성 — 직원은 통화만 하면 녹음이 안전하게 자동 보관된다. 한글 파일명 등 전송 오류 해결 및 안정화. 전체 아키텍처와 다음 단계(자동 전사·분석) 계획 확정.

## 2026-06-06 (토)
- **평판 시스템 — 병원명 인식 정확도**: 일반 단어("우리" 등)와 겹치는 짧은 별칭이 엉뚱한 글을 병원 언급으로 오인식하던 문제를 차단하고, 원인·해결을 운영 문서로 정리.

## 2026-06-05 (금)
- **평판 시스템 — 대응 관리 본격화**: 악성 게시물에 대한 정보삭제 요청서를 실무 양식으로 자동 작성하고 PDF로 발급(공문 자동화), 발신 정보 자동 보강, 대응 플레이북(법률 자문 반영) 설계. AI 분류 작업 큐를 **위험·부정 글 우선** 처리로 재정비. 서버 워커 **자가치유 감독 체계** 도입으로 반복 다운을 근본 해결. 고객 리포트를 전체 병원·실데이터 기반으로 개편. 이슈·트러블슈팅 기록(문제기록) 체계 신설.
- **통화보안 앱 — 착수**: 통화녹음 자동 수집을 검증하고 클라우드 업로드까지 동작 확인. 비용과 자동 전사 엔진 분석.

## 2026-06-04 (목)
- **평판 시스템**: 분석 워커가 몇 시간마다 멈추던 문제의 **근본 원인을 최종 확정하고 해결**. 검색 모니터링을 (제목·미리보기만이 아니라) **본문까지 수집**하는 방식으로 전환 — 제목은 멀쩡하지만 본문에서 불만이 드러나는 글을 놓치던 구조적 한계를 해소. 별칭 오매칭(어미·아이돌 이름·타 병원·뉴스)을 '무관'으로 자동 분류. 작업일지 자동 생성 체계 신설.

## 2026-06-03 (수)
- **평판 시스템 — 장애 근본 추적**: 분석 서버가 6시간 주기로 멈추는 문제를 하루 종일 추적. 메모리 부족, 캐시 누적, 재시작 정책 등 여러 가설을 하나씩 **데이터로 검증·기각**한 끝에, 진짜 원인(서버가 스스로 정상 종료한 컨테이너를 인프라가 다시 켜주지 않음)을 찾아 근본 해결. 원인 추적을 위해 워커 생사를 DB에 기록하는 계기 장치를 도입.

## 2026-06-02 (화)

## 2026-06-01 (월)
